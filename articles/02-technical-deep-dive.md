# 深度解析 AI Helper 的 ReAct 推理循环架构

> 适合平台：SegmentFault、V2EX、博客园、掘金技术区  
> 关键词：ReAct、推理循环、Chrome 扩展架构、多智能体、工具调度  
> 阅读时间：约 12 分钟

---

## 前言：当我们说"浏览器 AI Agent"时，我们在说什么？

市面上大多数"AI 浏览器插件"的工作模式是：获取页面文本 → 拼到 Prompt 里 → 调用 API → 显示回复。这本质上是一个带网页上下文的聊天框，它只是"看了"网页内容。

真正意义上的 **AI Agent** 需要具备三个核心能力：

1. **感知**：理解页面状态和用户意图
2. **决策**：自主决定需要调用什么工具、以什么顺序调用
3. **行动**：实际执行操作并获取反馈

AI Helper 的 ReAct 推理循环就是围绕这三个能力设计的。这篇文章将深入剖析其架构设计和实现细节。

---

## 一、ReAct 推理循环：不止是"调用 API"

### 1.1 传统模式 vs ReAct 模式

```
传统对话模式：
User: "帮我把这个页面存为 PDF"
LLM: "抱歉，我无法操作浏览器"

ReAct 模式：
User: "帮我把这个页面存为 PDF"
LLM → 思考: "需要截图并导出为 PDF"
      → 调用工具: capture_page()
      → 工具返回: { screenshot_data: "base64..." }
      → 调用工具: download_file(data, "page.pdf")
      → 工具返回: { success: true }
      → 回答: "页面已保存为 page.pdf"
```

### 1.2 循环核心流程

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ 用户输入  │───→│ LLM 推理     │───→│ 工具决策      │
└──────────┘    └──────────────┘    └──────┬───────┘
                                           │
                    有工具调用？              │
                    ↙        ↘              │
                  是          否             │
                  ↓           ↓             │
          ┌──────────┐  ┌──────────┐        │
          │ 工具执行  │  │ 返回答案  │        │
          └────┬─────┘  └──────────┘        │
               │                            │
               └──────── 结果反馈 ──────────→│
                                            │
               ┌────────────────────────────┘
               ↓
         继续推理循环
```

### 1.3 关键技术：工具预筛选

AI Helper 拥有 62 个内置工具，如果每次都将完整的工具列表发送给 LLM，会产生巨大的 Token 消耗。

**解决方案**：前置轻量 API 调用，用用户的原始问题对所有工具进行相关性评分，只将得分最高的 5-10 个工具注入到 Prompt 中。

```javascript
// 工具预筛选示意
const relevantTools = await preselectTools(userQuery, allTools, {
  maxTools: 8,
  minScore: 0.3,
  strategy: 'semantic-similarity'
});
```

这一优化在大多数场景下将 Token 消耗降低了 **60-80%**，同时因为减少了无关工具的干扰，任务完成质量反而有所提升。

## 二、Token 预算管理：把钱花在刀刃上

### 2.1 问题背景

在多轮推理循环中，每一轮的工具调用结果都会累积到上下文中。如果不对上下文大小加以控制，不仅 Token 成本失控，还会超出模型的上下文窗口导致推理中断。

### 2.2 动态预算策略

AI Helper 不简单地按"保留最近 N 条消息"来截断，而是基于模型上下文窗口和剩余 Token 预算动态决策：

```
预算模型：
总预算 = 模型上下文窗口 × 使用比例（可配置，默认 0.8）
已用 = 系统提示词 + 工具定义 + 历史消息 + 工具结果
剩余 = 总预算 - 已用

当剩余预算 < 阈值时：
  1. 压缩工具结果（只保留关键字段）
  2. 摘要化早期对话
  3. 如果仍不足，触发上下文压缩
```

## 三、并行工具执行

### 3.1 为什么需要并行

很多工具调用之间没有依赖关系。例如：

```
用户："帮我打开这三个网站：GitHub、掘金、V2EX"
```

传统串行执行需要 3 轮推理循环。AI Helper 支持工具声明 `parallelizable: true`，同一轮中的独立工具调用通过 `Promise.all` 并发执行。

### 3.2 依赖感知调度

系统通过工具依赖声明自动判断执行顺序：

```javascript
// 工具定义中的依赖声明
{
  name: 'fill_form',
  dependencies: ['get_page_text'],  // 先读取页面，再填表单
  parallelizable: false
}
```

## 四、多级超时与取消机制

### 4.1 三级超时控制

| 超时级别 | 默认值 | 作用 |
|----------|--------|------|
| API 超时 | 120s | 单次 LLM API 调用 |
| 工具超时 | 30s | 单个工具执行（敏感操作 30s 自动拒绝） |
| 循环超时 | 300s | 整体推理循环（可配置） |

### 4.2 取消机制

用户可随时取消推理循环。取消机制按会话隔离，不会影响其他会话的进行中的任务。取消后，已完成的部分结果会保留在对话历史中。

## 五、三级反思系统：让 AI 自己给自己打分

这是 AI Helper 区别于大多数 AI 工具的关键设计：

### Level 1：工具级反思

每次工具调用后，评估结果的有效性：
- 提取失败？尝试不同的选择器
- 数据为空？检查页面是否正确加载

### Level 2：子任务反思

对于分解后的子任务，评估：
- 结果是否完整？
- 是否与原始子任务目标一致？

### Level 3：后置反思（7 维度评分）

最终答案生成后，进行一次全面的质量评估：

| 维度 | 权重 | 检查内容 |
|------|------|----------|
| 完整性 | 高 | 是否回答了所有问题？ |
| 准确性 | 高 | 信息是否正确？有无幻觉？ |
| 相关性 | 中 | 是否紧扣用户意图？ |
| 工具使用 | 中 | 工具调用是否恰当？ |
| 清晰度 | 中 | 回答是否易于理解？ |
| 安全性 | 低 | 有无泄露敏感信息？ |
| 效率 | 低 | 是否存在冗余步骤？ |

综合评分低于阈值时，自动触发修订或重新推理。

## 六、架构设计的几个亮点

### 6.1 无框架状态管理

AI Helper 没有引入 React/Vue 等框架。状态管理采用 **双导出 Proxy 模式**，所有模块共享同一份响应式状态对象，避免了框架引入的体积开销和复杂性。

### 6.2 Content Script 消息路由

Content Script 中通过 Map 实现 O(1) 消息路由查找，而非大量 if/else 或 switch/case。

### 6.3 Agent 服务的安全设计

Node.js 代理服务的安全分层：

- **路径沙箱**：`realpath` 解析防符号链接绕过，白名单目录
- **命令安全**：黑名单（rm -rf / 等）、灰名单（需确认）、白名单（非敏感命令）
- **配对认证**：4 位动态码 + extensionId 双重验证
- **审计日志**：JSON Lines 格式，按日归档，所有操作可追溯

## 七、总结

AI Helper 的 ReAct 推理循环不是一个简单的"调 API → 返回结果"的管道，而是一个精心设计的 AI Agent 运行时。从工具预筛选到 Token 预算管理，从并行执行到多级反思，每个环节的设计目标都是：**用最少的 Token，完成最可靠的任务**。
